# 自动化减负：如何用一次输入，在多机与容器里自动查清「风险包」是否被引用

**机械化，自动化的终极目的是为了人类的自由，人不应该被困在无意义的重复上**

> 大多数人以为自动化的意义是"省时间"。但省下来的时间去哪儿了？如果只是躺在那里，你只是换了一种方式浪费它。自动化的真正价值，是让你有时间去做那些时间越充裕越有价值的事。

很多人问：自动化到底值不值得做？我的答案很简单：**只要这件事会重复出现，就值得自动化。**

在我的观念里，如果一件事，可预见还会有重复执行的情况，输入输出可固化，就应该交给自动化，AI时代来临前，自动化的成本可能会有点高，包括写脚本，写代码，调试。这个时候还需要预估一下，写脚本的时间和重复的时间的预期，比如可能重复一两次的事情，就没必要固化或者随便写个临时的脚本算了。AI 发展到今天，我现在可以暴力的认为，只要输入输出明确，就应该固化成自动化，无论是脚本，App, 或者是dify,n8n,coze这类工作流本质都一样。都是把确定性的内容交给脚本,API，把不确定但是可以模糊预处理的部分交给AI，最终决策交给人。

自动化的产品有很多，我以前接触过，自己写过的脚本也有很多，但是很多都没有被推广出来，我总结的问题是

1. 需要用户复杂的操作
2. 内部逻辑不够健全
3. 往往都当成临时脚本，没有考虑留存

我希望用一个个小的故事告诉你 ，自动化这件事到底要怎么做，哪些事情可以自动化，自动化带来了什么好处。

最近我做了一个「包引用排查脚本」，就是一个很最基础的例子。它用到的技术很简单，但他起到的作用很明显。

场景很常见：

- 安全团队或用户说扫描到一批风险问题，有的客户可能会给扫描路径有的不会。
- 你要快速回答：哪些机器用了它？是文件存在，还是进程正在引用？
- 还有容器环境、jar 打包、文件删了但进程还在跑这些边界情况

如果靠手工做，通常是这样：

1. 登录一台机器  
2. 跑几条命令  
3. 复制结果  
4. 再登录下一台  
5. 再重复一次

如果产品线包含十几台机器，你要扫描十来个包，那这个操作可能要重复100次。

---

## 自动化怎么设计

针对一个场景，我们的目的一定先是用起来够简单，如果用起来过于复杂，就失去了自动化的意义。把复杂留给自己，把简单送给客户。

这次脚本设计，我只坚持三个原则：

### 1) 输入尽量简单

用户只要提供：

- 目标范围（IP段 / IP列表 / IP文件）
- 要检查的文件名关键字

不要求用户记太多参数，也不要求先懂脚本内部逻辑。

### 2) 扫描尽量低开销

默认只做最有价值的检查：

- 文件名命中
- 运行进程命中
- Docker 内命中

把高成本扫描（比如内容深扫）放成可选项，而不是默认全开。

### 3) 报告尽量直接可用

报告不堆统计噪声，直接给排查人员最关心的明细：

- 哪台机器命中了
- 命中的文件路径
- 命中的进程 PID / 路径 / 启动命令

目标很明确：**看完就能决策，不需要二次翻译**。

## 脚本与仓库入口

这次场景使用的是“包引用排查脚本”，用于快速定位某个目标包在多台机器上的文件命中、进程命中和容器命中情况。

我会把这个脚本和后续所有自动化实践统一放在一个仓库里，便于持续维护和复用：

- 脚本目录占位：`scripts/package-reference-scan/`
- GitHub 仓库地址：`[GitHub 仓库地址待补充]()`

---

## 这个场景脚本的大体逻辑

为了让流程稳定可复用，这个脚本的核心逻辑是固定的：

1. 接收输入目标（IP段 / IP列表 / 文件）和包名关键字  
2. 批量连接目标主机，执行低开销基础扫描  
3. 分别采集文件命中、进程命中、Docker 容器命中  
4. 聚合为统一结构化结果，输出机器维度明细  
5. 生成可读报告，供排查人员快速判断  
6. 由人工做最终复核与处置决策

---

## 自动化真正解决了什么

很多人把自动化理解为“省几分钟命令时间”，其实这只是表层。

它真正解决的是：

- 降低人为漏查（机器不会忘记某台主机）
- 降低结果口径不一致（脚本输出是统一格式）
- 降低上下文切换成本（人不用反复做机械动作）
- 提升响应速度（从“半天收集”变成“分钟级产出”）

一句话：**自动化把“体力活”变成“脑力活”**。

---

## 人应该把时间花在哪

自动化 + AI 的组合，不是为了“不要人”，而是为了让人做更重要的事。

我认为工程师最该投入时间的，是下面这些：

- 判断结果是否可信
- 判断风险是否可接受
- 判断处理优先级是否正确
- 判断这次问题能否沉淀成长期机制

也就是说，**人只关注最重要的部分：验证与决策**。
