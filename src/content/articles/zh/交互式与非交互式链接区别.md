大家好，今天记录一个自动化相关的生产问题。从问题定位，问题复现，根因分析，问题修复，长期优化的角度完整的记录整体的解决思路，希望以后能帮到遇到同样问题的朋友。

------

## 一、问题背景

### 1. 故障现象

用户一台 **2C Linux 服务器**频繁触发 CPU 使用率过高告警：

- 多个**bash 进程**持续占用 CPU，单进程约 50%，合计超过80%，整体CPU超过90%产生告警；

### 2. 初步定位

通过进程信息 + 作业日志 + 时间线比对，锁定问题来自自动化平台，bash进程数量与自动化作业数量一致。

通过检查作业发现：

- 执行的都是简单脚本 / 命令（如 `top -n1`），无计算密集逻辑；
- 高 CPU 时间点与自动化作业打印执行远程连接日志基本一致。

问题初步定位为：ssh连接导致的CPU进程升高，且持续时间很短。

连接代码：

```
channel = ssh.invoke_shell(term="dumb", width=2048)
```

通过搜索资料和AI后发现可能是因为`invoke_shell` 会分配伪终端 PTY，PTY比较占用资源导致。

为确保结论可靠，我们剥离业务代码，**用最小脚本复现问题**。

------

## 二、问题复现

### 复现思路

- 本机 Python 控制远程 SSH；
- 远程采集 `bash/sshd` CPU；
- 分别测试 `invoke_shell(PTY)` 和 `exec_command(非PTY)`；
- 统计峰值 CPU 并对比。

### 复现脚本（见文章末尾）

- 本机：`ssh_pty_cpu_test.py`（控制端 + 分析）
- 远程：内嵌 `/tmp/remote_monitor.sh`（监控采集）

### 复现结论

- **PTY 模式**：bash 进程 连接时CPU 峰值极高，平均 40%+，连接完成后，执行命令时回落；
- **非 PTY 模式**：CPU 占用极低，通常 <5%。

------

## 三、根因分析：为什么 PTY 如此耗 CPU？

原始代码使用：

```python
channel = ssh.invoke_shell(term="dumb", width=2048)
```

### 核心原因

`invoke_shell()` 会**分配伪终端 PTY**，使远程 bash 进入**交互式模式**：

- 完整加载 `.bashrc/.profile`、命令历史、提示符、行编辑；
- 对超长命令行、多环境变量、UTF-8 字符做复杂解析；
- 即使只执行一条简单命令，也会走全套初始化流程。

### CPU 飙升链路

1. `invoke_shell` → 服务端分配 PTY
2. 发送长命令 → bash 启动
3. **交互式初始化 + 命令解析 → CPU 暴增**
4. 初始化完成 → 执行具体命令，CPU 回落到具体命令的消耗上

------

## 四、解决方案

### 1. 临时解决方案

用户当前无交互式自动化作业需求，直接切换连接模式：

**从 PTY → 非 PTY**

```python
channel = ssh.get_transport().open_session()
channel.set_combine_stderr(True)

env_cmd = (
    "source /etc/profile; "
    "[ -f ~/.bash_profile ] && source ~/.bash_profile; "
    "unset PROMPT_COMMAND; unset PS1; "
    f"cd {remotePath}; {remoteCmd}; exit $?"
)
channel.exec_command(env_cmd)
```

**效果**：CPU 立即下降，告警消失。

------

### 2. 长期解决方案

#### 探针化执行（产品原设计方案）

- 被管机部署**自动化作业探针**；
- 作业本地执行，**彻底去掉 SSH 开销**；
- 交互式指令通过专用 session 通道下发。

#### 智能连接模式（用户不接受安装探针）

- 执行前**判断是否需要交互**；
- 普通脚本 / 命令：**非 PTY**
- 含 sudo/top 等交互：**PTY**
- 实现最小化 CPU 开销 + 最大兼容。

------

## 五、交互式 vs 非交互式 SSH 对比

|  对比项  |  交互式 PTY (invoke_shell)   | 非交互式 (exec_command) |
| :------: | :--------------------------: | :---------------------: |
| PTY 分配 |              有              |           无            |
| CPU 占用 |              高              |          极低           |
| 环境加载 |    全量 .bashrc/.profile     |        基础环境         |
| 适用场景 | 交互式命令，比如需要用户输入 |   批量脚本、常规命令    |
| 生产推荐 |         仅必要时使用         |       自动化首选        |

------

## 六、非交互式模式避坑

1. `top` 必须用批处理：`top -b -n1`
2. `pgrep` 避免自匹配：`pgrep -f jav[a]`
3. 环境变量需显式加载，不依赖 `.bashrc`
4. `sudo/passwd` 等交互不适用非 PTY

------

## 七、可能存在的问题（交互式 vs 非交互式 SSH 执行）

我们在从 `invoke_shell`（交互式）转向 `exec_command`（非交互式）的过程中，发现两种方式在行为上存在明显差异。虽然非交互式执行更轻量、资源占用低，但也带来了一些实际使用中的坑，总结如下：

### 1. 非交互式执行无法支持需要终端的命令

比如执行：

```
ssh root@192.168.140.74 'top -n1'
```

会报错：

```
TERM environment variable not set.
```

这是因为 `top` 这类基于终端的交互式程序，依赖 `TERM` 环境变量和终端能力（如光标控制、清屏等）。在非交互式模式下，SSH 不分配伪终端（PTY），`top` 拿不到终端信息，直接退出。

✅ **解决方法**：

- 显式加上 `-b` 批处理模式：`top -b -n1`
- 或设置 `TERM` 环境变量：`TERM=dumb top -b -n1`

> 所以不是 `top` 不能用，而是要用对模式。

### 2. 命令中的子命令执行时机容易出错

典型例子：

```
ssh root@192.168.140.74 'sh t1.sh "$(pgrep -n -f java)"'
```

这里我们本意是想在**远程服务器上**执行 `pgrep`，获取最新一个 Java 进程的 PID，然后传给 `t1.sh`。

但如果不加引号，或者引号使用不当，`$(pgrep ...)` 可能会在**本地 shell** 就被展开，导致执行的是本地的 `pgrep`，结果完全错误。

即使加了引号，也存在另一个问题：

SSH 在远程执行命令时，实际上是这样运行的：

```
bash -c 'sh t1.sh "$(pgrep -n -f java)"'
```

而 `pgrep -n -f java` 是按名字模糊匹配的，它有可能会把 `bash -c ...` 这个命令本身也匹配进去（因为命令行里有 `java` 关键字），导致 `pgrep` 返回的是它自己这条命令的 PID，而不是真正的 Java 服务。

这就造成了 **PID 获取不准**的问题。

✅ **解决方法**：

- 使用 `[j]ava` 这类技巧绕过自身匹配：

```
ssh root@192.168.140.74 'sh t1.sh "$(pgrep -n -f jav[a])"'
```

这样 `pgrep` 命令本身是 `jav[a]`，不会匹配到 `java` 字面，避免了自匹配。

- 或者更稳妥：在远程用脚本封装逻辑，避免复杂内联命令。

### 3. 环境变量和特殊字符处理更敏感

非交互式执行时，远程 shell 通常是非登录、非交互式的 `sh` 或 `bash -c`，不会加载 `.bashrc`、`.profile` 等文件。

这意味着：

- 一些你习惯的别名、路径、环境变量可能不存在
- 中文字符、特殊符号在某些系统上可能编码异常
- 脚本中依赖的 `PYTHONPATH`、`JAVA_HOME` 等需要显式设置

✅ **建议**：所有依赖的环境变量，都应在命令中明确导出，不要依赖远程用户的默认环境。

### 4. 无法进行多轮交互

这是最明显的限制。像 `sudo`、`vim`、`passwd`、`mysql` 等需要多次输入的命令，在非交互式模式下无法正常工作。

比如：

```
ssh root@192.168.140.74 'sudo service restart nginx'
```

如果需要输入密码，就会卡住或直接失败。

## 八、附录：复现验证脚本

### 本机脚本：ssh_pty_cpu_test.py

```python
import paramiko
import time
import pandas as pd
from datetime import datetime

HOST = "127.0.0.1"
USER = "ubuntu"

# 远程监控脚本路径（全部放在 /tmp）
REMOTE_SCRIPT = "/tmp/remote_monitor.sh"

# 远程监控脚本内容（内嵌在Python里，自动上传）
MONITOR_SCRIPT_CONTENT = '''#!/bin/bash
OUT=$1
echo "ts,pid,ppid,tty,pcpu,cmd" > "$OUT"
end=$((SECONDS + 5))

while [ $SECONDS -lt $end ]; do
    ts=$(date +%Y-%m-%dT%H:%M:%S.%3N)
    ps -eo pid,ppid,tty,pcpu,cmd | grep -E 'bash|sshd' | grep -v grep | \\
    awk -v ts="$ts" '{printf "%s,%s,%s,%s,%s,\\"%s\\"\\n", ts,$1,$2,$3,$4,substr($0,1,80)}'
    sleep 0.01
done >> "$OUT"
'''

def upload_remote_script(ssh):
    """上传监控脚本到远程 /tmp"""
    sftp = ssh.open_sftp()
    with sftp.file(REMOTE_SCRIPT, 'w') as f:
        f.write(MONITOR_SCRIPT_CONTENT)
    ssh.exec_command(f"chmod +x {REMOTE_SCRIPT}")
    sftp.close()
    time.sleep(0.2)

def cleanup_remote_files(ssh, log_remote_list):
    """清理：监控脚本 + 本次生成的所有CSV"""
    csv_files = " ".join(log_remote_list)
    ssh.exec_command(f"rm -f {REMOTE_SCRIPT} {csv_files}")
    time.sleep(0.2)

def get_remote_time(ssh):
    _, stdout, _ = ssh.exec_command("date +'%Y-%m-%d %H:%M:%S.%3N'")
    return stdout.read().decode().strip()

def single_test(is_pty, test_id):
    ssh = paramiko.SSHClient()
    ssh.set_missing_host_key_policy(paramiko.AutoAddPolicy())
    ssh.connect(HOST, username=USER, timeout=5)

    # 日志放在 /tmp
    log_remote = f"/tmp/test_{test_id}.csv"

    # 启动监控
    ssh.exec_command(f"nohup {REMOTE_SCRIPT} {log_remote} >/dev/null 2>&1 &")
    time.sleep(0.5)

    # 测试 PTY / 非 PTY
    if is_pty:
        chan = ssh.invoke_shell(term="dumb")
        time.sleep(0.3)
        chan.send("exit\n")
    else:
        chan = ssh.get_transport().open_session()
        chan.exec_command("sleep 0.3")
        chan.makefile("rb").read()

    time.sleep(4)

    # 下载日志
    sftp = ssh.open_sftp()
    log_local = f"temp_{test_id}.csv"
    sftp.get(log_remote, log_local)
    sftp.close()
    ssh.close()

    # 分析CPU
    df = pd.read_csv(log_local)
    df["pcpu"] = pd.to_numeric(df["pcpu"], errors="coerce").fillna(0)

    if is_pty:
        target_df = df[df["cmd"].str.contains("-bash", na=False)]
    else:
        target_df = df[df["cmd"].str.contains("sshd", na=False)]

    return target_df["pcpu"].max() if not target_df.empty else 0.0

# ===================== 主实验流程 =====================
if __name__ == "__main__":
    print("=" * 80)
    print("    🧪 SSH PTY CPU 开销验证实验")
    print("=" * 80)

    # 先建立连接上传监控脚本
    print("\n📤 上传监控脚本到远程 /tmp ...")
    init_ssh = paramiko.SSHClient()
    init_ssh.set_missing_host_key_policy(paramiko.AutoAddPolicy())
    init_ssh.connect(HOST, username=USER, timeout=5)
    upload_remote_script(init_ssh)
    init_ssh.close()

    results = []
    remote_logs = []  # 记录所有远程CSV，用于最后清理

    # 5次测试
    for i in range(1, 6):
        print(f"\n🔹 第 {i} 次实验...")
        pty_cpu = single_test(is_pty=True, test_id=f"pty_{i}")
        time.sleep(1)
        npty_cpu = single_test(is_pty=False, test_id=f"npty_{i}")
        time.sleep(1)

        results.append({
            "实验次数": i,
            "PTY模式峰值CPU(%)": round(pty_cpu, 1),
            "非PTY模式峰值CPU(%)": round(npty_cpu, 1)
        })
        remote_logs.append(f"/tmp/test_pty_{i}.csv")
        remote_logs.append(f"/tmp/test_npty_{i}.csv")

    # 生成结果表
    df_results = pd.DataFrame(results)
    stats = {
        "实验次数": "平均值",
        "PTY模式峰值CPU(%)": round(df_results["PTY模式峰值CPU(%)"].mean(), 1),
        "非PTY模式峰值CPU(%)": round(df_results["非PTY模式峰值CPU(%)"].mean(), 1)
    }
    df_final = pd.concat([df_results, pd.DataFrame([stats])], ignore_index=True)

    print("\n" + "="*80)
    print("    📊 实验结果")
    print("="*80)
    print(df_final.to_markdown(index=False))

    # 结论
    avg_pty = df_results["PTY模式峰值CPU(%)"].mean()
    avg_npty = df_results["非PTY模式峰值CPU(%)"].mean()

    print("\n" + "="*80)
    print("    🎯 验证结论")
    print("="*80)
    print(f"✅ PTY 模式平均CPU：{avg_pty:.1f}%")
    print(f"✅ 非PTY 模式平均CPU：{avg_npty:.1f}%")
    print(f"✅ PTY 开销 = 非PTY的 {avg_pty/max(avg_npty, 0.1):.1f} 倍")
    print("\n💡 验证结果：SSH PTY 交互式模式会显著消耗CPU，非PTY几乎无开销。")
    print("="*80)

    # ===================== 自动清理远程文件 =====================
    print("\n🧹 正在清理目标端所有临时文件...")
    clean_ssh = paramiko.SSHClient()
    clean_ssh.set_missing_host_key_policy(paramiko.AutoAddPolicy())
    clean_ssh.connect(HOST, username=USER, timeout=5)
    cleanup_remote_files(clean_ssh, remote_logs)
    clean_ssh.close()
    print("✅ 目标端清理完成：/tmp/remote_monitor.sh + 所有CSV已删除")
```
